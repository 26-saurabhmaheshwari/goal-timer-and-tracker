# time-dashboard

Single-file personal time/goal/habit dashboard. **The entire app is `index.html`** (~2200 lines, embedded CSS + JS, Chart.js via CDN). It runs from `file://` and deploys as one file to GitHub Pages.

## Hard constraints (violating these breaks things)
- **Never split `index.html`.** The user edits the `file://` copy directly; index.html is the source of truth, not a build artifact. No `src/` + build step.
- **Never use a preview server / headless Chrome** to render it — isolated filesystem, wasted tokens. The user eyeballs on `file://` and reports back.
- **Never touch Firestore Security Rules.** The user pastes those manually.
- **FILE MAP comment at top of file** must contain NO nested `-->` (a nested comment terminator breaks the whole page).
- Edit > Write. Don't re-read the whole file to verify an edit.

## State model & the recurring bug classes
Persisted state = localStorage keys (`LS_*` consts ~line 766) + in-memory mirrors (`COLORS`, `ICONS`, `ALLCATS`, `TASK_CAT`, `MONTHS`, ...). Firestore overrides localStorage when signed in.

**Merge precedence (the "color/icon doesn't persist" bug).** Startup merge MUST be `{...SEED, ...loadObj(KEY, {})}` — seeds are defaults, user-saved values override. The reverse (`...SEED` last) silently reverts user edits on every load. `test.js` guards this.

**Don't cache `loadEntries()` in a module var.** Callers do "load → mutate in place → conditionally save" and `applyBlob` writes `LS_KEY` directly (bypassing `saveEntries`), so a shared-reference cache leaks unsaved mutations and goes stale after a cloud pull/import. The dataset is tiny (raw entries purged at 3 months), so the `JSON.parse`-per-paint cost is negligible — not worth the correctness risk. Considered and rejected 2026-07.

**Adding any new persisted state (simplified 2026-07):**
1. `LS_<NAME>` const + `SEED_<NAME>` if it has defaults, in-memory init with correct merge precedence, load/save helpers, persist fn calling `localStorage.setItem` + `fbSyncPush()`.
2. **Register one row in `SYNC_FIELDS`** (`// --- Firebase sync glue ---` section): `[blobField, lsKey, liveGetter, mergeStrategy]`. That single row wires cloud push (`fbBlob`), cloud restore (`applyBlob` via `fb-data`), AND the Settings→Personal Export/Import backup. dob/name/target/planwin/lifebar are scalar inline specials in fbBlob/applyBlob; `LS_VIEW` is intentionally unsynced.
`test.js` asserts every declared `LS_*` const is in `SYNC_FIELDS` or the exempt list — run it with `node test.js` (Node v24 LTS installed 2026-07-17; if the shell can't find `node`, refresh PATH from Machine+User env first).

**Sync merge (added 2026-07).** Cloud pull (`fb-data`) calls `applyBlob(d,true)` = per-field merge so two devices don't clobber each other; file import calls `applyBlob(d)` = wholesale replace (backup restore). `mergeStrategy` per row: `'id'` union arrays by `.id` cloud-wins-on-conflict (entries, goals), `'map'` `{...local,...cloud}` (colors, icons, taskcat, rollup, online), `'union'` dedupe-append string array (allcats), `'replace'` cloud wins (months, habits — habits have no stable id). Tradeoff: a delete on one device can reappear from the other (no tombstones) — accepted over the old whole-blob overwrite that silently dropped a device's offline edits. Map/union strategies read the **in-memory mirror** via `liveGetter` (e.g. `()=>COLORS`), not localStorage. **No sign-in allowlist** — any Google account signs into its own `users/{uid}` doc; per-user isolation is enforced by Firestore Security Rules (user-maintained).

**Planning rows are ALLCATS-driven.** `YTARGETS` was deleted (2026-07) — `planYearTotals()` builds one row per Settings category (plus extras for logged/goal/habit cats not in ALLCATS), filters out all-zero rows. Targets are always computed from goals+habits; never reintroduce a hardcoded target list.

**Category data is Settings-driven, never hardcoded.** Icons/colors/names all come from the Categories tab (`ALLCATS`/`COLORS`/`ICONS`), resolved via `iconFor()` / `colorFor()`. Don't reintroduce hardcoded icon/color maps in render code.

**Seeds are neutral, not personal (added 2026-07).** All `SEED_*` (colors/icons/allcats/taskcat/goals/habits/months/online) are generic starter data (Health/Learning/Work/Relationships/Creativity, "Read 12 books", etc.), `SEED_MONTHS`/`SEED_ONLINE` empty. Seed goal dates use `SEED_YR=CUR_YM.slice(0,4)`, never a hardcoded year. Don't reintroduce personal/religious seeds. First-run onboarding = `#onboard` banner on Overview, gated by `maybeShowOnboard()`: shows only when `LS_ONBOARDED!=='1'` AND no saved data (`LS_GOALS`/`LS_KEY`/`LS_ALLCATS` all absent). `LS_ONBOARDED` is device-local UI state — exempt from `SYNC_FIELDS` (like `LS_VIEW`), listed in test.js exempt set.

## Layout conventions
- Modal: `.modal-ov` overlay + `.card.modal-card`, toggled via `hidden` + body overflow.
- Sidebar-nav pages (Settings) use `.tipswrap` / `.tipside` / `.tipmain`.
- PDF export = `window.print()` with `@media print` hiding `.noprint`/nav/tabs; `hidden` views don't print.
- **Export surfaces (consolidated 2026-07, #38):** exactly three — Reports→Export PDF (`repPdfBtn`, the only whole-dashboard print entry; the old topbar PDF button was deleted), period-modal "🖨 Print & share" dropdown (`#peShare`, a `details.repdrop` holding `#pePrint`/`#pePoster`/`#peIcs`/`#peWitness`; wiring goes through `peAct(id,fn)` which closes the menu before running — new items go inside the panel + a `peAct` call, not as loose footer buttons), and Settings→Backup ("⬇ Download backup"/"⬆ Restore backup", still ids `expBtn`/`impBtn`). `window.print()` calls inside generated print-window HTML (paper sheet, poster) are part of those flows, not separate surfaces.
- Goals have `sec`: `'active' | 'backlog' | 'dream'`.
- Goal cards are slim by default: the 4 `gd-row` detail rows (`.gcard-det`) render `hidden` and open via a `details ▾` pill (`.gdet-btn` in `.gcard-foot`) → `goalFlip(id)` (flip-in animation, one card open at a time, pure DOM so re-render closes it). 📜 Log is an icon in `.gcard-acts` next to ✎ (no standalone row).
- Period-entry + pledge modals use one-line `.perow` rows (name ellipsizes via `.penm`, `.pein` inputs, `.pe-z` zebra on odd index, `.pehead` column-header row, `.peamt`+`.unitlab` for amount+unit) — don't reintroduce the 2-line wrapping `.frow` layout there.
- **7 top-level tabs** (`#viewNav`, added 2026-07 tab-diet): overview / goals / planning / log / pot / reports / settings. **No standalone Audit or FAQ tab** — Audit (`#viewAudit`) is a card that sits directly below `#viewLog` in the DOM and shows whenever `v==='log'` (rendered by `renderLog()` calling `renderAudit()`); FAQ is the `setFaq` section inside the Settings sidebar (`#setSide` button `data-s="faq"`). `setView()` remaps stale persisted `LS_VIEW` values `'audit'→'log'` and `'faq'→'settings'` at the top, so old bookmarks still resolve.

## Goal log, mark-complete, planning window (added 2026-07)
- **Per-goal log** = `goal.logs[]` (nested prop, rides on `LS_GOALS` + Firebase sync, no new key). Each row `{ym, period, amt, total}` where `total` = the cumulative goal total AFTER that addition. Never purged; gone only when the goal is deleted. **Both progress paths funnel through one helper `addGoalProgress(g, x, amt, ym, period, extraLog)`** (bumps `x.cur`, pushes the log row, writes the tracked-time entry when `goalMins>0`, returns mins): `submitGoal()` (Log Data tab → 🎯 Goal — main path, no `extraLog`) and the goal card's `+` Add-progress modal (`addForm` submit, passes `{d}` as `extraLog`). If you add a third way to bump `goal.cur`, call `addGoalProgress` — don't re-implement it, or the log silently under-counts.
- **Log display** = a `📜 Log (n)` button on the card → `goalLogShow(id)` opens the `#logModal` popup (scales to 100s; card stays compact). Row renders `prev +amt → total unit`, newest first (`prev = total - amt`).
- **Mark complete** = `goal.done` (bool). Toggle via `goalToggleDone(id)`; the button lives in the **edit modal** footer (`#gDone`, shown/labelled in `goalEditStart`), NOT on the card. Done goals: hidden from active/backlog/dream filters, shown in `#secCompleted`, excluded from `renderGoalStats`, `paceLabel` returns 'Done', card gets `.gcard-done`, `+` hidden. Logged time is retained; reopen sets `done=false`.
- **Soft-delete + Trash** (added 2026-07) = `goal.deleted` (bool) + `goal.deletedAt` (ISO). `goalDel(id)` now soft-deletes (sets the flags, KEEPS the linked time entries); `goalRestore(id)` clears them; `goalPurge(id)` is the old hard-delete (removes goal + its `type:'goal'` entries). **Display/math paths use `goalsLive()` (`loadGoals().filter(g=>!g.deleted)`); save/lookup paths keep using `loadGoals()`** so trashed goals survive round-trips (drag-reorder, toggleDone, category-rename, form save all `loadGoals()→mutate→saveGoals`). Trash renders in `#secTrashWrap`/`#secTrash` (whole card hidden when empty) via `renderTrash()`. Rides the existing `LS_GOALS` blob → syncs/exports with no new key. If you add a NEW goal display/math site, route it through `goalsLive()` or trashed goals reappear.
- **Periods as life currency** (added 2026-07) = pure display, no new state. Life bar (`renderLifeBar`, opt-in) has a "N ten-day periods left" chip = `round(leftDays/10)`. Goal cards show a "≈ N periods to finish" `det` row via `goalPeriodsToFinish(x)` = observed pace (`cur / days-since-start`) projected onto the remaining amount, `/10`, ceil; returns null (row hidden) when not started, no `start`, `cur<=0`, or done. Derived from existing dob/target/pace math — never persisted.
- **Next-period pledge** (added 2026-07) = `goal.commit={ym,period,amt}` (nested prop on the goal, rides `LS_GOALS`, no new key). After `savePeriodEntry()` succeeds it calls `openCommitStep(savedYm,savedPeriod)` → the `#commitModal` overlay renders one optional number input per active goal for the NEXT period (`nextPeriod(ym,p)`, mirror of `prevPeriod`), pre-filled from any existing matching `commit`. `saveCommits()` writes `commit` for amounts >0 and clears it when blanked for that period. The pledge surfaces two places: `renderPeriodEntry` goal row (🎯 badge + ✅/⚠️ actual-vs-pledge once that period is being logged) and the printed paper sheet goal row. Skip/close leaves goals untouched. Only ONE commit per goal (the latest pledge) — pledging a new period overwrites the old.
- **Wall poster + period verdict** (added 2026-07) = `periodVerdict(ym,period[,ents,perDay])` scores one 10-day block against the **fixed-habit plan** (`Σ active-habit m/day × days-in-period`): returns `'hit'`(≥80% logged) / `'partial'`(≥40%) / `'miss'`(<40%) for elapsed periods, `'future'` for ones not yet finished with no logs, `'none'` if no habit plan. Logged = Σ `e.mins` where `pkey(e)` matches (all entry types count, so goal time helps). Uses CURRENT habits to judge past periods (no habit history — same tradeoff as habit-adherence). `printYearPoster()` opens a new window (same self-contained-HTML + auto-`window.print()` pattern as `printPeriodSheet`) rendering a 12-month × 3-period grid: elapsed cells pre-coloured by verdict, future cells blank to hand-colour. Button `#pePoster` sits next to `#pePrint` in the period modal. `savePeriodEntry()` appends the just-saved period's verdict colour to its flash. **No new persisted state** — verdict is derived on the fly.
- **Letter to future self** (added 2026-07) = one synced object `LS_LETTER='ptd_letter_v1'` = `{text, writtenYm}`. `loadLetter()`/`saveLetter(text)` (save stamps `writtenYm=CUR_YM` + `fbSyncPush`). Registered in `SYNC_FIELDS` as `['letter', LS_LETTER, loadLetter, 'replace']` (single object, cloud-wins wholesale) — test.js's LS_* guard requires this row. Written via a textarea + `#setLetterSave` button in Settings→Personal (populated by `renderPersonalSettings`). Surfaces on the Overview `#letterCard` via `maybeShowLetter(nudge)`: shows only when a non-empty letter exists AND not dismissed this year AND (`nudge` true OR it's Dec 31+). The nudge passed on every Overview render is `mostGoalsBehind()` = `>half` of active targeted goals are Behind/Lagging by `paceLabel`. `renderLetterCard()` fills the text + a "You wrote this in {Mon YYYY}" line from `writtenYm`. **Dismiss is device-local & unsynced**: `letterDismiss()` writes `LETTER_SEEN='ptd_letterseen_v1'` = the current year — key deliberately NOT prefixed `LS_` so test.js ignores it (no SYNC_FIELDS/exempt entry needed). Tradeoff: one dismiss hides the card for the whole calendar year, including the Dec 31 trigger — accepted as the gentler, non-nagging choice.
- **Witness note** (added 2026-07) = pure display, no new state. `periodWitness(ym,period)` returns a plain-text **3-line** string to send an accountability partner: L1 `📮 {plabel} — {verdict}` (reuses `periodVerdict`), L2 what got done (sum of `goal.logs` amounts matching this ym+period, per goal `name +amt unit`, top 4 + "+N more"), L3 the next-period pledge (`goal.commit` matching `nextPeriod(ym,period)`, top 4). `copyWitness(ym,period,msgEl)` copies via `navigator.clipboard`; on failure (blocked on `file://`) it drops the text into `msgEl` to copy by hand. Button `#peWitness` in the period-modal footer summarizes the currently-selected `peMonth`/`pePeriod`. Derived entirely from existing goal logs/commits — never persisted.
- **Renounced list** (added 2026-07) = `LS_RENOUNCED='ptd_renounced_v1'` = array of `{id, text}` (deliberate no's — things given up to protect time for goals). `loadRenounced`/`saveRenounced`, `renAdd(text)`/`renDel(id)` (id via `eid()`), `renderRenounced()` fills `#renList` (strike-through rows + ✕). Registered in `SYNC_FIELDS` as `['renounced', LS_RENOUNCED, loadRenounced, 'id']` (union-by-id across devices).
