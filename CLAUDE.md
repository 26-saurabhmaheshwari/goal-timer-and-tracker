# time-dashboard

Single-file personal time/goal/habit dashboard. **The entire app is `index.html`** (~2200 lines, embedded CSS + JS, Chart.js via CDN). It runs from `file://` and deploys as one file to GitHub Pages.

## Hard constraints (violating these breaks things)
- **Never split `index.html`.** The user edits the `file://` copy directly; index.html is the source of truth, not a build artifact. No `src/` + build step.
- **Never use a preview server / headless Chrome** to render it вЂ” isolated filesystem, wasted tokens. The user eyeballs on `file://` and reports back.
- **Never touch Firestore Security Rules.** The user pastes those manually.
- **FILE MAP comment at top of file** must contain NO nested `-->` (a nested comment terminator breaks the whole page).
- Edit > Write. Don't re-read the whole file to verify an edit.

## State model & the recurring bug classes
Persisted state = localStorage keys (`LS_*` consts ~line 766) + in-memory mirrors (`COLORS`, `ICONS`, `ALLCATS`, `TASK_CAT`, `MONTHS`, ...). Firestore overrides localStorage when signed in.

**Merge precedence (the "color/icon doesn't persist" bug).** Startup merge MUST be `{...SEED, ...loadObj(KEY, {})}` вЂ” seeds are defaults, user-saved values override. The reverse (`...SEED` last) silently reverts user edits on every load. `test.js` guards this.

**Don't cache `loadEntries()` in a module var.** Callers do "load в†’ mutate in place в†’ conditionally save" and `applyBlob` writes `LS_KEY` directly (bypassing `saveEntries`), so a shared-reference cache leaks unsaved mutations and goes stale after a cloud pull/import. The dataset is tiny (raw entries purged at 3 months), so the `JSON.parse`-per-paint cost is negligible вЂ” not worth the correctness risk. Considered and rejected 2026-07.

**Adding any new persisted state (simplified 2026-07):**
1. `LS_<NAME>` const + `SEED_<NAME>` if it has defaults, in-memory init with correct merge precedence, load/save helpers, persist fn calling `localStorage.setItem` + `fbSyncPush()`.
2. **Register one row in `SYNC_FIELDS`** (`// --- Firebase sync glue ---` section): `[blobField, lsKey, liveGetter, mergeStrategy]`. That single row wires cloud push (`fbBlob`), cloud restore (`applyBlob` via `fb-data`), AND the Settingsв†’Personal Export/Import backup. dob/name/target/planwin/lifebar are scalar inline specials in fbBlob/applyBlob; `LS_VIEW` is intentionally unsynced.
`test.js` asserts every declared `LS_*` const is in `SYNC_FIELDS` or the exempt list вЂ” run it with `node test.js` (Node v24 LTS installed 2026-07-17; if the shell can't find `node`, refresh PATH from Machine+User env first).

**Sync merge (added 2026-07).** Cloud pull (`fb-data`) calls `applyBlob(d,true)` = per-field merge so two devices don't clobber each other; file import calls `applyBlob(d)` = wholesale replace (backup restore). `mergeStrategy` per row: `'id'` union arrays by `.id` cloud-wins-on-conflict (entries, goals), `'map'` `{...local,...cloud}` (colors, icons, taskcat, rollup, online), `'union'` dedupe-append string array (allcats), `'replace'` cloud wins (months, habits вЂ” habits have no stable id). Tradeoff: a delete on one device can reappear from the other (no tombstones) вЂ” accepted over the old whole-blob overwrite that silently dropped a device's offline edits. Map/union strategies read the **in-memory mirror** via `liveGetter` (e.g. `()=>COLORS`), not localStorage. **No sign-in allowlist** вЂ” any Google account signs into its own `users/{uid}` doc; per-user isolation is enforced by Firestore Security Rules (user-maintained).

**Planning rows are ALLCATS-driven.** `YTARGETS` was deleted (2026-07) вЂ” `planYearTotals()` builds one row per Settings category (plus extras for logged/goal/habit cats not in ALLCATS), filters out all-zero rows. Targets are always computed from goals+habits; never reintroduce a hardcoded target list.

**Category data is Settings-driven, never hardcoded.** Icons/colors/names all come from the Categories tab (`ALLCATS`/`COLORS`/`ICONS`), resolved via `iconFor()` / `colorFor()`. Don't reintroduce hardcoded icon/color maps in render code.

**Seeds are neutral, not personal (added 2026-07).** All `SEED_*` (colors/icons/allcats/taskcat/goals/habits/months/online) are generic starter data (Health/Learning/Work/Relationships/Creativity, "Read 12 books", etc.), `SEED_MONTHS`/`SEED_ONLINE` empty. Seed goal dates use `SEED_YR=CUR_YM.slice(0,4)`, never a hardcoded year. Don't reintroduce personal/religious seeds. First-run onboarding = `#onboard` banner on Overview, gated by `maybeShowOnboard()`: shows only when `LS_ONBOARDED!=='1'` AND no saved data (`LS_GOALS`/`LS_KEY`/`LS_ALLCATS` all absent). `LS_ONBOARDED` is device-local UI state вЂ” exempt from `SYNC_FIELDS` (like `LS_VIEW`), listed in test.js exempt set.

## Layout conventions
- Modal: `.modal-ov` overlay + `.card.modal-card`, toggled via `hidden` + body overflow.
- Sidebar-nav pages (Settings) use `.tipswrap` / `.tipside` / `.tipmain`.

## Goal log, mark-complete, planning window (added 2026-07)

- **Log display** = a `рџ“њ Log (n)` button on the card в†’ `goalLogShow(id)` opens the `#logModal` popup (scales to 100s; card stays compact). Row renders `prev +amt в†’ total unit`, newest first (`prev = total - amt`).
- **Mark complete** = `goal.done` (bool). Toggle via `goalToggleDone(id)`; the button lives in the **edit modal** footer (`#gDone`, shown/labelled in `goalEditStart`), NOT on the card.
